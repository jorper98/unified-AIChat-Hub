# Feature: Global System Prompt with Date/Time & Weather Injection

## Overview

Add a **Global System Prompt** that is automatically injected into every chat session, separate from per-thread instructions. This prompt will include dynamic context (current date/time, weather) that models don't inherently know about. This is the foundation for adding tools/skills to every chat.

## Architecture

The current system has **per-thread system instructions** stored in thread metadata. The new **Global System Prompt** is stored in the `settings` collection and applies to ALL conversations. They serve different purposes:

- **Global System Prompt** (NEW): Always-injected context — date/time, weather, location, and any persistent instructions you want the model to always know
- **Per-Thread System Instruction** (EXISTING): Conversation-specific behavior — "You are a helpful coding assistant", role-playing, task-specific instructions

Both are prepended to the message history before sending to the model.

## Changes Required

### 1. New Utility: `src/lib/context.ts` (CREATE)

A server-side utility that generates dynamic context at request time:

```typescript
export async function buildSystemContext(): Promise<string>
```

This function:
- Gets the current date and time in the configured timezone
- Optionally fetches weather from wttr.in (free, no API key needed) based on `WEATHER_LOCATION` env var
- Returns a formatted context string like:
  ```
  Current date and time: Wednesday, June 3, 2026 at 7:19 PM PDT.
  Current weather in San Diego, CA: Partly cloudy, 72°F (22°C), wind 5 mph.
  ```

**Design decisions:**
- Uses `wttr.in` for weather — free, no API key, simple text output. If `WEATHER_LOCATION` is not set, weather is omitted.
- Uses `Intl.DateTimeFormat` for timezone-aware date formatting. Default timezone from `TIMEZONE` env var (e.g., `America/Los_Angeles`), falls back to UTC.
- Weather fetch has a 3-second timeout to avoid blocking chat responses.
- No caching — each request gets fresh data (weather APIs are fast and rate limits are generous).

### 3. Modify `src/app/api/chat/route.ts`

Inject the global system prompt + dynamic context as the FIRST system message:

After line 121 (`const db = await getDb();`), add:
1. Fetch the global system prompt from settings
2. Call `buildSystemContext()` to get date/time/weather
3. Prepend a combined system message before the per-thread instruction

The message construction logic (before the existing `if (systemInstruction)` block at line 173):

```typescript
// Build dynamic context (date/time, weather)
const dynamicContext = await buildSystemContext();

// Fetch global system prompt from settings
const settings = await db.collection('settings').findOne({ _id: 'global_settings' as any });
const globalPrompt = settings?.globalSystemPrompt || '';

// Build the global system message
if (dynamicContext || globalPrompt) {
  let globalMessage = '';
  if (dynamicContext) {
    globalMessage += dynamicContext + '\n\n';
  }
  if (globalPrompt) {
    globalMessage += globalPrompt;
  }
  formattedHistory.unshift({ role: 'system', content: globalMessage.trim() });
}
```

This ensures the global context is injected BEFORE any per-thread instruction. The model sees:
1. System: "Current date/time + weather + your global instructions"
2. System: "Your per-thread instruction (if any)"
3. User/Assistant messages...

## How This Enables Future Tools/Skills

This architecture establishes the pattern for injecting server-side context into every chat. Future tools/skills can follow the same pattern:

- **File access**: Inject file contents or directory listings into the system prompt
- **Database queries**: Pre-fetch relevant data and include it as context
- **Web search**: Run a search and inject top results before the model responds
- **Code execution**: Add a code runner and inject results

For full OpenAI-style function calling (where the model decides when to call tools), a larger refactor would be needed:
1. Define tool schemas in the API request (`tools` array)
2. Handle `tool_calls` in the model response
3. Execute the tool server-side
4. Send results back as `tool` role messages
5. Loop until the model stops requesting tool calls

The current approach (pre-injected context) is simpler, works with all models, and is sufficient for date/time/weather.

## Risks & Considerations

- **Weather API latency**: wttr.in adds ~200-500ms per request. The 3-second timeout prevents hanging. If weather fails, chat continues without it.
- **Token usage**: The injected context adds ~50-100 tokens per request. Negligible for most models.
- **No per-model customization**: The global prompt applies to all models. This could be extended later if needed.
